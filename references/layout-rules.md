# 레이아웃 최적화 규칙 (Layout Rules)

> SKILL.md 워크플로우 ④ 단계에서 적용하는 자동 점검·교정 규칙.
> Claude는 빌드 직전 모든 콘텐츠에 이 규칙을 통과시킨 뒤 HWPX로 변환합니다.

---

## 1. 한 문장 = 한 줄 원칙 (단일 줄 규칙)

### 1-1. 기준선

- **본문 한 줄 = 약 35–45자** (한글 기준, 기본 본문 폭 42,520 HWPUNIT, 11pt 글꼴 기준)
- **35자 이하**: 안전
- **35–45자**: 자동 검토 대상 (사용자에게 표시)
- **46자 이상**: **자동 분리 후보** (반드시 처리)

### 1-2. 분리 규칙

긴 문장은 다음 우선순위로 분리:

1. **`및` / `,(쉼표)` 기준 분리** — 동등 항목 나열일 경우 가장 자연스러움
2. **종결어미 변환** — `~하고, ~하며, ~한 후` 같은 연결어를 종지부로
3. **부사절 분리** — `~으로 인해, ~에 따라`로 끊고 다음 문장 시작
4. **개조식 항목화** — 한 문장이 2개 이상의 개념을 담고 있으면 `-` 글머리로 분할

예시:

❌ 라마단 종료에 따라 중동항로의 거래량과 적재율 회복이 예상되며, 라마단 직전 적재율은 95% 수준이었고, 선사협의체는 성수기 할증료 부과를 유예하였습니다. (66자, 3개 정보 혼재)

✅
- 라마단 종료 → 중동항로 거래량·적재율 회복 예상
- 라마단 직전 적재율: 약 95%
- 성수기 할증료(USD 300/TEU) 부과 유예

## 2. 페이지 걸침 방지

### 2-1. 문단·섹션 단위로 끊기

**한 문단이 두 페이지에 걸쳐 분리되면 가독성이 크게 떨어집니다.**

다음 단위는 **반드시 같은 페이지에 유지**해야 합니다:

- 같은 □ 대제목 아래의 첫 ○ 본문 (제목과 첫 항목은 함께)
- 표 헤더와 첫 행
- 그림 캡션과 그림
- 인용 박스 안의 내용
- 1페이지 보고서의 모든 내용 (이름 그대로 1페이지 안에 들어가야 함)

### 2-2. 자동 처리

Claude 는 매핑 전 콘텐츠 길이를 추정해 다음 처리:

1. **페이지당 권장 라인 수 산출 — 양식별 여백이 다르므로 값이 다르다**
   - **풀버전** (여백 상30·하25mm, 11pt): **약 38–42줄/쪽**
   - **1페이지 보고서** (여백 상25·하20mm): **실효 상한 약 37줄** — 38–42줄을 그대로 쓰면 1p 에서 과대 허용이 되어 페이지를 넘길 수 있으므로 반드시 37줄 기준을 쓴다
   - 시행문 (여백 상40·하30mm): 별도 산정 (1–3쪽 허용이라 라인 게이트보다 별첨 분리 권고 우선)
2. **섹션 단위 분리 시뮬레이션** — 각 □ 섹션의 라인 수 추정
3. **페이지 걸침 발생 시 사용자 확인** — 두 가지 옵션 제시:
   - (A) 강제 페이지나누기 삽입
   - (B) 콘텐츠 압축 후 1페이지 유지
4. **1페이지 보고서 양식**의 경우, 초과 시 **단축 제안**을 표시하고 빌드 거부

### 2-3. 단축 제안 알고리즘

1페이지 양식에서 콘텐츠가 초과할 때:

1. 부가설명(`*`로 시작하는 라인)을 우선 제거 후보로 표시
2. 비고/참고 항목을 후순위로 표시
3. 동의어 압축(`~할 것으로 보입니다` → `~ 예상`) 자동 적용
4. 조사·접미사 정리(`-적`/`-의`/`-것`/`-들`) 자동 적용
5. 그래도 초과하면 풀버전 보고서로 양식 변경 권고

## 3. 글머리 위계 일관성

### 3-1. 두 체계 중 택일

**체계 A: 행정·실무용 (1페이지 보고서·메모형)**

```
□ 대제목 (16pt, 굵게)
  ○ 본문 항목 (15pt)
    - 세부 항목 (14pt)
      * 부가 설명 (13pt, 회색)
        ※ 비고·주의 (13pt)
```

**체계 B: 정식 행정문서 (시행문·풀버전 보고서)**

```
Ⅰ. 추진배경
  1. 환경분석
    가. 외부환경
      (1) 거시환경
        (가) 정책 동향
          ① 세부 항목
            ㉮ 더 세부
```

### 3-2. 혼용 금지

한 문서 안에서는 **한 체계만** 사용. 두 체계가 동시에 등장하면 Claude 가 매핑 전 경고·통일한다.

### 3-3. 양식별 기본값

| 양식 | 기본 위계 |
|------|-----------|
| 시행문 | 체계 B (1. → 가. → 1) → 가) → (1) → (가) → ① → ㉮) |
| 1페이지 보고서 | 체계 A (□ → ○ → - → *) |
| 풀버전 보고서 | 체계 B (Ⅰ. → 1. → 가. → (1) → (가)) |
| 이메일 | 체계 A 단순화 (□ → - → *) |

## 4. 숫자·날짜 강조 정렬

### 4-1. 숫자는 본문 흐름에서 분리

| 패턴 | 처리 |
|------|------|
| 본문 안의 단순 수치 | `매출 5,210억원` (천단위 콤마, 단위 축약) |
| 비교 가능한 수치 | 표로 분리, 헤더 우측 `(단위: 천원)` 표기 |
| 핵심 KPI | 굵게 + 색상 (파란색 절제) |
| 단일 날짜 | `2025. 1. 5.` (마침표 뒤 공백, 0 패딩 없음) — 행안부 공문서 편람 |
| 날짜 범위 | `2025. 1. 5. ~ 2025. 3. 31.` (단일 날짜 규범 + `~` 양옆 공백 1칸) |
| 시간 | `15:00` (24시간제) |

### 4-2. 표기 통일 규칙

- 천 단위 구분: 항상 콤마(`5,210`), 공백 금지
- 단위는 한글로 표기(원·천원·억원·억·%·㎡)
- 영문 단위는 단위명 사이 공백 1칸(`USD 300`, `KRW 5,000`)
- 비율은 소수점 1자리 또는 정수(`13.9%`, `116%`), 정밀도 혼용 금지
- **날짜는 0 패딩하지 않는다** (`1. 5.` ✓ / `01. 05.` ✗). 범위도 단일 날짜와 동일 규범.
  압축형(`2025.01.05`)은 **표 셀·괄호 안 등 공간 제약 시에만** 예외 허용.

## 5. 제목·소제목 단축 규칙

### 5-1. 자동 단축 후보

다음 패턴이 보이면 Claude 가 매핑 전 단축 제안:

| 원형 | 단축 |
|------|------|
| `~와 관련된 ~` | `~ 관련 ~` 또는 단순 `~ ~` |
| `~에 대한 ~` | `~ ~` 또는 동사형 |
| `~을 위한 ~` | `~용 ~` 또는 단순 `~ ~` |
| `~에 따른 ~` | `~ 따른 ~` 또는 `~ 후 ~` |
| `~의 ~의 ~` | `~ ~ ~` (의 연쇄 제거) |

### 5-2. 명사형 제목 변환

| 동사형 (긴 제목) | 명사형 (짧은 제목) |
|------------------|-------------------|
| 중동항로 거래량이 회복될 가능성에 대한 검토 | 중동항로 거래량 회복 전망 |
| 라마단 종료에 따른 중동항로 동향 분석 | 라마단 후 중동항로 동향 |
| 매장 운영의 효율성을 제고하는 방안 | 매장 운영 효율화 방안 |

규칙: **동사 → 명사화 / 부사절 → 형용절 / `~의 ~의 ~`는 한 단어로 압축**.

## 6. 시각 스타일 (참조 파일이 없을 때 기본값)

> 사용자 답변: **공공기관 표준에 가까운 무채색·중립 스타일**

### 6-1. 색상 팔레트

| 용도 | 색상 | HEX | 비고 |
|------|------|-----|------|
| 본문 텍스트 | 검정 | `#000000` | 기본 |
| 보조 텍스트 | 짙은 회색 | `#404040` | 부가설명·비고 |
| 강조 (제한적) | 짙은 남색 | `#1F3864` | 핵심 KPI·소제목만 |
| 표 헤더 배경 | 옅은 회색 | `#D9D9D9` | 패턴 통일 |
| 표 강조행 | 매우 옅은 회색 | `#F2F2F2` | 합계행 등 |
| 음영 박스 (요약) | 옅은 회색 | `#F2F2F2` | 1p 요약문 박스 |

### 6-2. 글꼴

- **본문**: 함초롬바탕 11pt (한글 표준)
- **제목**: 함초롬돋움 16pt 굵게
- **소제목**: 함초롬돋움 13–14pt 굵게
- **표·캡션**: 함초롬바탕 10–11pt
- **부가설명**: 함초롬바탕 10pt 회색

> 환경에 함초롬 글꼴이 없으면 `맑은 고딕 / 한컴바탕`으로 폴백.

## 7. 양식 미준수 시 처리

레이아웃 규칙을 위반한 결과물은 다음 절차로 처리:

1. **빌드 전 자동 점검** — Claude 가 매핑 전 모든 위반 사항을 리포트
2. **사용자에게 표 형태로 위반 내역 제시** — 수정 가능 항목 / 수동 검토 필요 항목 분리
3. **자동 수정 가능 항목**은 적용 후 사용자에게 "Before/After" 비교 표시
4. **수동 검토 필요 항목**은 표시한 채로 빌드 진행 (※ 표시)

## 8. 자동 적용하는 변환 목록 (Claude가 매핑 *전* 적용)

> v3.4.0 부터 이 변환들은 별도 스크립트가 아니라 **Claude 가 콘텐츠를 슬롯에 매핑하기
> 전에 직접 적용**하는 작성 원칙입니다.

| # | 패턴 | 변환 | 신뢰도 | 장르 |
|---|------|------|--------|------|
| 1 | `~와 관련된` | `~ 관련` | 높음 | 전체 |
| 2 | `~에 대한` | (생략) 또는 동사형 | 중 | 전체 |
| 3 | `~할 것으로 보입니다` | `~ 예상` | 높음 | **1p·풀버전만** |
| 4 | `~한 것으로 판단됩니다` | `~ 판단` | 높음 | **1p·풀버전만** |
| 5 | `~할 예정이었으나 이를 유예하였습니다` | `~ 예정 → 유예` | 높음 | **1p·풀버전만** |
| 6 | `~하는 것이 필요합니다` | `~가 필요합니다` | 높음 | 전체 |
| 7 | `~ 중 하나인` | (생략) 또는 `대표적인` | 중 | 전체 |
| 8 | `여러 ~들 / 많은 ~들 / 각 ~들 / 모든 ~들` | `여러 ~ / 많은 ~ / 각 ~ / 모든 ~` | 높음 | 전체 |
| 9 | `사회적 / 경제적 / 정치적 / 행정적 / 조직적` (명사 직전) | `사회 / 경제 / 정치 / 행정 / 조직` | 중 | 전체 |
| 10 | `~의 ~의 ~` (의 연쇄) | `~ ~ ~` (의 1개 이하) | 높음 | 전체 |
| 11 | 한 문장 46자 초과 | 분리 후보로 표시 | — | 전체 |
| 12 | 영문 약어 + 한글 사이 띄어쓰기 누락 | 띄어쓰기 추가 | 높음 | 전체 |

신뢰도 **높음** = 자동 적용. **중** = 사용자 확인 후 적용.

### 8-1. ⚠️ 장르 가드 — 명사형 종결 변환은 시행문·이메일에 미발동

패턴 **#3·#4·#5** 는 서술어 종결(`~보입니다`/`~판단됩니다`/`~하였습니다`)을
**명사형 종결(개조식)** 로 바꾼다. 이 변환은 **1페이지 보고서·풀버전 보고서에서만** 발동한다.

**시행문(공문)·이메일에는 발동하지 않는다.** 시행문 본문은 서술어 완결 + 공손체
(`~하시기 바랍니다`)가 규범이고(→ [`format-gongmun.md`](format-gongmun.md) 결문 어조),
이메일도 인사말·완결 문장이 기본이므로, 명사형 종결로 바꾸면 격식 위반이 된다.

- **전체 발동(#1·2·6~12)**: 군더더기 제거(적/의/것/들·관련·대한)와 띄어쓰기 교정은
  서술어 종결을 건드리지 않으므로 4개 양식 모두 적용.
- **1p·풀버전만(#3·4·5)**: 서술어를 명사형으로 압축하는 변환은 개조식 규범인 두 양식에만.

> 시행문·이메일에서 문장이 길면, 명사형 압축 대신 **문장 분리(#11)** 로 처리한다.
